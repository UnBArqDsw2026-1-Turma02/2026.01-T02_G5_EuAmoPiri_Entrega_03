# 3.1.3 Factory Method

## Introdução

Dentre os padrões criacionais,o Factory Method se destaca por tratar da forma como os objetos são instanciados no sistema. Em vez de criar objetos diretamente por meio de chamadas explícitas aos construtores, o padrão propõe a utilização de um método responsável por encapsular essa criação.

Segundo Craig Larman, a responsabilidade de criação deve ser atribuída de forma a reduzir o acoplamento e aumentar a coesão. Nesse contexto, o Factory Method delega a criação de objetos a uma classe específica, permitindo que o sistema trabalhe com abstrações em vez de depender diretamente de implementações concretas.

## Objetivo

Aplicar o padrão de projeto Factory Method (GoF) no sistema, com o objetivo de centralizar a criação de objetos e reduzir o acoplamento entre as classes.
A implementação busca permitir que diferentes tipos de entidades sejam instanciados a partir de uma mesma interface, delegando essa responsabilidade a uma classe fábrica. Dessa forma, pretende-se tornar o sistema mais flexível, facilitando sua manutenção e possibilitando a adição de novos tipos de objetos sem impactar o código cliente.

## Metodologia

A aplicação do padrão Factory Method (GoF) foi realizada em dois contextos distintos do sistema: na criação de objetos do tipo Local e de Interações. A escolha desses pontos se deu pela necessidade de instanciar diferentes tipos de objetos a partir de um mesmo fluxo de dados, provenientes do frontend.

### Factory Method para local

O Factory Method foi aplicado na criação de objetos do tipo Local, que possuem variações como **PontoTuristico** e **Estabelecimento**.

O objetivo dessa implementação foi centralizar a lógica de criação desses objetos, evitando a utilização de múltiplas estruturas condicionais (if/else) espalhadas pelo sistema. Dessa forma, a responsabilidade de decidir qual tipo concreto deve ser instanciado passa a ser da classe LocalFactory.
Essa abordagem permite que o sistema trabalhe com a abstração Local, reduzindo o acoplamento com classes específicas e facilitando a adição de novos tipos de locais no futuro, sem a necessidade de alterações no código da aplicação.

## Implementação

### Abstração e Classes concretas

```
class Local(ABC):
    def __init__(self, nome, descricao, endereco):
        self.nome = nome
        self.descricao = descricao
        self.endereco = endereco

    @abstractmethod
    def exibir_informacoes(self):
        pass

```

```
class PontoTuristico(Local):
    def exibir_informacoes(self):
        return f"Ponto Turistico: {self.nome} - {self.descricao}"

```

```
class Estabelecimento(Local):
    def exibir_informacoes(self):
        return f"Estabelecimento: {self.nome} - {self.descricao}"
```

A classe **Local** define uma abstração comum, enquanto **PontoTuristico** e **Estabelecimento** representam implementações concretas, garantindo flexibilidade e reutilização.

### Factory Method com Criadores

```
class CriadorLocal(ABC):

    @abstractmethod
    def factory_method(self, **kwargs):
        pass
```

```
class CriadorPonto(CriadorLocal):
    def factory_method(self, **kwargs):
        return PontoTuristico(**kwargs)
```

```
class CriadorEstabelecimento(CriadorLocal):
    def factory_method(self, **kwargs):
        return Estabelecimento(**kwargs)
```

O padrão Factory Method é aplicado por meio da classe **CriadorLocal**, que define o método de criação, e suas subclasses, que determinam qual objeto concreto será instanciado.

### Classe **LocalFactory**

```
class LocalFactory:
    _tipos = {
        "ponto_turistico": CriadorPonto,
        "estabelecimento": CriadorEstabelecimento,
    }

    @staticmethod
    def criar_local(tipo, **kwargs):
        classe = LocalFactory._tipos.get(tipo.lower())

        if not classe:
            raise ValueError(f"Tipo '{tipo}' não é suportado.")

        return classe().factory_method(**kwargs)
```

A **LocalFactory** atua como um ponto central de criação, mapeando o tipo informado para o criador correspondente, permitindo maior organização e extensibilidade.
